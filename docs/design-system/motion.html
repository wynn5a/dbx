Foundations

Motion & Animation

Motion in Data Buddy is fast, quiet and functional — it confirms an action, reveals structure, or shows where something came from, then gets out of the way. Nothing loops or bounces for decoration. If an animation doesn't help the user understand state, it doesn't ship.

Principles
01Fast over fluid

Most transitions land in 80–180ms. The interface should feel instant; motion is a hint, not a performance. When in doubt, make it shorter.

02Ease out, not in

Things arrive quickly and settle gently. Default to an ease-out curve so motion decelerates into rest — never linear, never ease-in for entrances.

03Move with meaning

Animate to explain: a panel rises from where it opened, a caret turns to show it expanded. No movement on idle, no parallax, no attention-seeking.

Duration scale

Five steps, each tied to a job. Reach for the smallest one that reads. The bars run in real time — --dur-2 (120ms) is the default control transition and the legacy --speed.

Easing

Four curves cover everything. --ease is the everyday default; --ease-out gives entrances and overlays a decisive arrival; --ease-in-out drives anything that moves continuously; --ease-spring adds a barely-there overshoot — reserved for toggles and checks.

Animate cheaply

The DS uses opacity + transform for entrances, never height animation — animating height or width is janky and forces layout. Reveal with a fade, or a clip, instead.

Rule of thumb

Animate transform and opacity. Both are GPU-composited and never trigger reflow — so motion stays at 60fps even on a dense grid.

Performance · transform + opacity only
Looping & status motion

The only motion allowed to run continuously — and only while something is genuinely pending or live. The moment the state resolves, the loop stops.

Rules
Do
  • Animate transform and opacity — they composite on the GPU and never reflow.
  • Default to ease-out so motion decelerates into rest.
  • Keep micro-interactions 80–180ms; reserve 240ms+ for overlays and full views.
  • Let one motion explain one change — open from where it was triggered.
  • Stop every loop the instant its state resolves.
  • Honor prefers-reduced-motion — drop movement, keep the final state and instant fades.
Don't
  • Don't move or scale on hover — tint the background, hold the position.
  • Don't animate height, width, top or left — they force layout and stutter.
  • Don't add bounce or big springs; the one allowed overshoot is the toggle knob.
  • Don't run decorative loops — no idle parallax, no breathing gradients, no marquee.
  • Don't stagger long lists or delay content the user is waiting on.
  • Don't cross 360ms for anything inside a working view.
Accessibility

Every animation is wrapped so it degrades to its end-state when the user asks for less motion. Movement and loops are removed; near-instant fades remain so state changes are still legible.

@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
  }
}
